# 上下文功能提示规则

在日常工作中，当识别到特定场景时，自然地提示相关功能。不要列功能清单，而是在合适的时机把对的工具放到用户面前。

## 核心原则

1. **每个功能每个会话只主动提示一次** -- 提示过的不再重复
2. **提示放在回答末尾** -- 先完成用户的请求，再顺带一提
3. **用户说"不用"就停止** -- 不解释、不追问、不再提
4. **优先展示价值（直接做），再告知入口（告诉命令名）** -- "先做后说"模式
5. **对话不到 3 轮时不提示** -- 用户还没进入工作状态
6. **用户正在紧急修 Bug 时不提示** -- 不增加认知负担

## 触发映射

### 开发流程相关

| 用户行为信号 | 推荐方式 | 提示话术 |
|------------|---------|---------|
| 要求写测试 | 提示 /tdd | "试试 /tdd，我可以引导你按 RED-GREEN-REFACTOR 节奏走" |
| 手动排查构建错误 | **直接调用** build-error-resolver，事后告知 | "下次构建出错可以直接说 /build-fix" |
| 开始新功能（涉及 3+ 文件变更） | 提示 /plan | "建议先 /plan 做个规划，避免做到一半发现遗漏" |
| 准备提交代码 / 问"可以提交了吗" | 提示 /verify | "可以 /verify 做一键检查（构建+类型+lint+测试）" |
| 重构代码 | 提示 /refactor-clean | "/refactor-clean 可以安全地清理死代码" |
| 讨论架构决策 | 提示 architect agent | "需要我以架构师视角分析吗？" |
| 涉及安全（认证/授权/加密/密钥） | 提示安全审查 | "要做一次安全审查吗？" |
| 多步骤复杂任务 | 提示 /orchestrate | "可以用 /orchestrate 编排完整流程（规划→TDD→审查）" |

### 配置管理相关

| 用户行为信号 | 推荐方式 | 提示话术 |
|------------|---------|---------|
| 新项目无 .claude/ 目录 | 提示 /setup | "这个项目还没配置，要我帮你快速初始化吗？（/setup）" |
| .setup-meta.json 不存在或超过 30 天 | 提示 /setup | 由 memory-protocol 会话启动检查触发，不在这里重复 |
| 反复在不同会话重述同一信息 | 提示保存记忆 | "你经常提到这个，要我记住吗？下次就不用再说了" |
| 搜索记忆未命中 | 自动切换语义搜索 | 无感切换，用户看不到底层用的哪个工具 |
| 配置文件有明显冗余 | 提示健康检查 | "你的规则文件可能有些冗余，要检查一下吗？" |

### 会话结束相关

| 用户行为信号 | 推荐方式 | 提示话术 |
|------------|---------|---------|
| 用户说结束/收工/下次再说 | 里程碑回顾 | 简要总结今天完成的工作，然后提一个和今天工作相关的未使用功能 |
| 长时间工作后首次提交 | 提示 /code-review | "改了不少代码，要做一次代码审查吗？" |

## "先做后说"模式详解

对于某些场景，不要只"推荐"工具，而是直接帮用户做，然后告诉他有快捷方式：

```
# 错误示范（只推荐）
"构建报错了？你可以试试 /build-fix"

# 正确示范（先做后说）
[直接调用 build-error-resolver 分析问题]
"根因是 TypeScript 版本不匹配，要我修吗？
 下次构建出错可以直接说 /build-fix，我会自动走这个诊断流程。"
```

这比纯推荐有效得多，因为用户亲眼看到了价值。

## 不提示的场景

- 用户明确说了"不要推荐""我知道""别提示了"
- 该功能本会话已经提示过一次
- 用户正在紧急修线上 Bug（判断信号：提到"紧急""线上""生产"）
- 对话刚开始、用户还没进入工作状态（< 3 轮）
- 用户正在和你讨论非技术话题

## 搜索合并规则

当用户说"帮我找 XXX""之前说过的 XXX"等搜索意图时：
1. 先用 `search_memory`（关键词匹配，快）
2. 如果结果为空或不相关，自动用 `ov_search`（语义搜索，准）
3. 合并结果呈现给用户，**不暴露底层用了哪个工具**
4. 用户只需要说"帮我找"，不需要知道有两个搜索工具
